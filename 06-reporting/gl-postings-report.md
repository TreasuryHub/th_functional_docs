# G/L Postings (Reporting)

> **Availability:** `Available` ✅
> **Where to find it:** Reporting › Operations › G/L Posting
> **Who uses it:** accountants, finance teams, controllers, auditors.
> **Permissions required:** `CashManagement.TransactionsPosting` · Read. See [Roles & Permissions](../00-getting-started/04-roles-and-permissions.md).

## Overview
The G/L Postings report shows how your transactions map to the general ledger. It's a **read-only
grid** of postings, each showing which ledger accounts are debited and credited and the rule that
generated it, so you can confirm that operational activity is being reflected correctly in
accounting.

This is the **reporting-side view** of postings — for the full accounting workflow (generating
journal entries, mapping the chart of accounts, and syncing to your ERP), see the accounting module.

## Key concepts
- **Posting** — the debit/credit entry generated for a transaction, mapping it to ledger accounts.
- **PostingRule** — the rule that generates the posting and determines which ledger accounts a given
  transaction posts to. Each posting shows its rule in the **Matching Rule** column, which links back
  to the rule that produced it.
- **Debit Acct / Credit Acct** — the two accounts of each entry.
- **Status** — where a posting stands (e.g. generated, posted to the ERP, pending, or a mismatch).

## Before you start
- Transactions must exist to have postings — see [Transactions](transactions.md).
- You need `CashManagement.TransactionsPosting` at **Read** to view this report.

## How to use it

### View postings
1. Open **Reporting › Operations › G/L Posting**.
2. The grid lists postings with these columns:

   | Column | Shows |
   |---|---|
   | **Booking Date** | The date the posting was booked. |
   | **Company** | The legal entity the posting belongs to. |
   | **Account** | The account the underlying transaction relates to. |
   | **Debit Acct** | The general-ledger account debited. |
   | **Credit Acct** | The general-ledger account credited. |
   | **Amount** | The posting amount. |
   | **Matching Rule** | The **PostingRule** that generated the posting (links to the rule). |
   | **Remittance Info** | The remittance message carried by the transaction. |
   | **Status** | Where the posting stands in its lifecycle. |

3. Sort any column to group your postings — for example all **Pending** postings together. The grid
   is read-only: postings are generated by their PostingRule, so there is no add or edit here.

### Choose columns and print
1. Use **Columns** to choose which columns are shown.
2. Use **Print** to prepare the current view for printing or handoff to your auditors.

## Tips & good practices
- Sort by **Status** to find postings that are pending or flagged as a mismatch, and resolve those
  first.
- Use this report to reconcile operations to the ledger — if a transaction isn't posting as expected,
  open its **Matching Rule** to check the PostingRule mapping.
- Print a period's postings for your auditors rather than granting broad system access.
